02

The Problem

How do you coordinate multiple AI agents
working on the same codebase?

🔀

Conflicts

Agents stepping on each other's work without coordination

👁

Visibility

No way to see what agents are doing or intervene mid-flight

🛑

Control

No emergency stop, no human override, no guardrails

03

The Solution

A sprint board that agents read & write,
with a dashboard for human control

Shared State

board.json is the single source of truth. All agents read it, claim tickets, and update status through it.

Blackboard Signals

blackboard.md lets agents broadcast findings, blockers, and decisions to each other.

Ticket State Machine

11 states with enforced transitions. No skipping steps. Branch per ticket. Reviewer merges.

Human Override

Move any ticket to any state. Halt all agents instantly. Full authority at all times.

04

Four Agents, Four Roles

Each agent runs in its own Swarm Agent session

PM
Planner
Reads the spec, grooms tickets, sets acceptance criteria and priorities
new → groomed
DEV
Builder
Claims tickets, creates feature branches, writes code and commits
dev-ready → review-ready
REVIEWER
Gatekeeper
Reviews PRs, requests changes or approves, merges to main
review-ready → test-ready
QA
Tester
Writes and runs tests, files bug tickets, marks tickets done
test-ready → done
05

Ticket Lifecycle

11 states, enforced transitions

new → groomed → dev-ready → in-dev → review-ready → in-review → test-ready → in-test → done
blocked changes-requested bug → new ticket
  • Human gate: groomed → dev-ready requires explicit approval
  • Branch per ticket: feat/TICKET-001-title — no direct main commits
  • Only reviewer merges PRs to main (or human override)
  • Bugs create new tickets linked to the parent

07

Multi-Project Support

Independent per-project agents — full isolation

📁

Per-Project Isolation

Each project gets its own Orchestrator, board, blackboard, sprint state, and agent logs. Switching projects doesn't stop agents on other projects.

🗺

Orchestrator Map

Server maintains Map<projectName, Orchestrator>. Each request resolves to the correct orchestrator via orchFor(req).

⚡

Cross-Project Rate Limiting

When any agent hits an API rate limit, ALL agents across ALL projects stop instantly. Account-level safety.

🚀

Create & Switch

Create projects via dashboard header or PM Chat. Switch with the project dropdown — agents keep running independently.

.agent-board/projects/
├── MyProject/       # board, blackboard, chat, logs
├── API-Service/    # independent agents
└── Mobile-App/     # independent agents

11

Safety & Recovery

Built-in guardrails at every step

🔒

Agent Guardrails

  • WIP limit of 1 ticket per agent
  • Must claim before working
  • Cannot skip states
  • No direct commits to main
  • Halt signal stops all work
🔄

Session Recovery

  • Agents crash? Auto-detected via board.json
  • Next invocation gets recovery prompt
  • Branch field is the recovery anchor
  • Agent checks out existing branch & resumes
  • No work lost, no duplicate branches
📋

Audit Trail

  • Every state change logged in ticket history
  • Append-only files in .agent-board/history/
  • Agent logs preserved per-session
  • Blackboard signals timestamped
🛡

Human Authority

  • Move ANY ticket to ANY status
  • Halt stops all agents instantly
  • Moving ticket warns only when agent is actively working
  • Confirmation dialog before destructive actions

Getting Started

Two ways to run the board

Option A: Dashboard (recommended)

cd dashboard
npm install
node server.js
# Open http://localhost:3456
# Click: Start Sprint
# Toggle: Auto-Dispatch ON
# Agents spawn automatically

Option B: Manual Terminals

# Terminal 1 — PM
swarm > /pm

# Terminal 2 — Dev
swarm > /dev

# Terminal 3 — Reviewer
swarm > /reviewer

# Terminal 4 — Test
swarm > /test
